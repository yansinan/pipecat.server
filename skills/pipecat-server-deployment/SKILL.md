# Pipecat Voice Server Deployment

Deploy a Pipecat voice agent with SmallWebRTC transport and the official PrebuiltUI client (pipecat-ai-prebuilt).

## Architecture

```
Browser (PrebuiltUI JS) ←HTTP→ FastAPI server (SmallWebRTC) ←pipeline→ STT → LLM → TTS → output
                                 │
                                 └── /start → return sessionId + iceConfig
                                 └── /sessions/{id}/api/offer (POST) → SDP exchange
                                 └── /sessions/{id}/api/offer (PATCH) → ICE candidates
                                 └── /client/ → PrebuiltUI static files
```

## Required packages

```toml
dependencies = [
    "pipecat-ai[webrtc,whisper]>=1.4.0",
    "pipecat-ai-prebuilt>=1.0.3",
]
```

## Common pitfalls (verified against pipecat 1.4.0 + pipecat-ai-prebuilt 1.0.3)

### 1. TransportParams class name

There is NO `SmallWebRTCTransportParams`. Use `TransportParams` from `pipecat.transports.base_transport`:

```python
from pipecat.transports.base_transport import TransportParams
# NOT from pipecat.transports.smallwebrtc.transport import SmallWebRTCTransportParams
params = TransportParams(audio_in_enabled=True, audio_out_enabled=True, ...)
```

### 2. /start response — any non-empty sessionId works; framework uses pc_id internally

The PrebuiltUI JS client (`@pipecat-ai/small-webrtc-transport`) uses `sessionId` only as
a URL path component (`/sessions/{sessionId}/api/offer`). The server-side framework
identifies connections by `pc_id` (generated by `SmallWebRTCConnection`), not by your
session ID. **No UUID generation needed.** Any non-empty string works:

```python
# Simplest: return a fixed placeholder. The client echoes it back in the URL,
# the server never reads it.
return {
    "sessionId": "default",   # or any fixed non-empty string
    "iceConfig": {"iceServers": [{"urls": ["stun:stun.l.google.com:19302"]}]},
}

# Empty string "" will 404 because the URL becomes /sessions//api/offer
# (FastAPI routes don't match the collapsed empty segment)
```

For internal logging use `connection.pc_id` (e.g. `SmallWebRTCConnection#0-abc123`),
which is shorter than UUID and maps directly to `webrtc_handler._pcs_map.keys()`.

### 3. Client uses /sessions/{id}/api/offer — register ONLY that path

**Old pitfall**: register both `/api/offer` and `/sessions/{session_id}/api/offer`. This
is wrong. `@pipecat-ai/small-webrtc-transport` (>=1.10) only ever calls the
`/sessions/{sessionId}/api/offer` path; the bare `/api/offer` is dead code from
the pre-1.10 protocol and is never reached by any current client.

**Correct**:
```python
@app.post("/sessions/{session_id}/api/offer")
async def offer_session(session_id, request, background_tasks):
    async def on_connection(connection):
        background_tasks.add_task(_run_pipeline, connection)
    return await webrtc_handler.handle_web_request(
        request=request, webrtc_connection_callback=on_connection,
    )

@app.patch("/sessions/{session_id}/api/offer")
async def ice_candidate_session(session_id, request: SmallWebRTCPatchRequest):
    return await webrtc_handler.handle_patch_request(request)
```

**Don't roll your own ICE handler.** `SmallWebRTCRequestHandler.handle_patch_request()`
already does `candidate_from_sdp` + `add_ice_candidate` for you — a custom
`_handle_ice_patch()` reproduces the framework's internals exactly.

### 4. pipeline runner must NOT block HTTP handler

Use `background_tasks.add_task()` to run the pipeline. Direct `await runner.run()` blocks the HTTP response and causes timeout:

```python
async def on_connection(connection):
    background_tasks.add_task(_run_pipeline, connection, session_id)

answer = await webrtc_handler.handle_web_request(
    request=request,
    webrtc_connection_callback=on_connection,
)
return answer  # HTTP returns immediately, pipeline runs in background
```

### 5. build_pipeline uses keyword-only args

```python
# Correct:
worker, ctx = build_pipeline(transport=transport)
# WRONG:
worker, ctx = build_pipeline(transport)  # TypeError: positional arg
```

### 6. ❌ DO NOT pass `observers=[RTVIObserver()]` — silent message drop

This pitfall was once thought correct but is now known to **silently drop all RTVI messages**.

```python
# ❌ WRONG — observer has no `rtvi` reference, drops every message
worker = PipelineWorker(
    pipeline,
    params=PipelineParams(...),
    observers=[RTVIObserver()],  # ← PipelineWorker detects external observer, skips default
)

# ✅ CORRECT — let PipelineWorker auto-create the observer wired to its RTVIProcessor
worker = PipelineWorker(
    pipeline,
    params=PipelineParams(...),
    # don't pass observers=
)

# ✅ With custom observer params:
worker = PipelineWorker(
    pipeline,
    params=PipelineParams(...),
    rtvi_observer_params=RTVIObserverParams(bot_output_transforms=[...]),
)
```

**Why** — `observer.py:205-220` initializes `self._rtvi = None` when no rtvi is passed.
`sender.send_rtvi_message()` checks `if self._rtvi:` and silently drops when None. PrebuiltUI
sees no BotOutput/BotLLMText/BotTTSStarted events → "Bot connected, but conversation panel
stays empty."

**Source**: official `pipecat-examples/code-helper/server/bot.py:188-198` uses the correct
pattern (only `rtvi_observer_params=`, never `observers=[...]`). When in doubt, copy
`code-helper` — it is the only official example that uses *separate* STT/LLM/TTS services.

### 6b. ❌ DO NOT put `RTVIProcessor()` in the `processors` list

`RTVIProcessor` and `RTVIObserver` are automatically created by `PipelineWorker` and
wired to each other. Adding `RTVIProcessor()` to the processors list causes one of:

1. `RTVIProcessor and RTVIObserver found, skipping default ones` — your external
   `RTVIObserver()` (without rtvi) silently drops messages (same trap as #6).
2. The default-observer guard fails: `RTVIProcessor found in pipeline but no
   RTVIObserver in observers` — pipeline refuses to start.

**Correct pipeline** (verified against `code-helper/server/bot.py:175-185`):
```python
processors = [
    transport.input(),
    stt,                # no RTVIProcessor() here
    user_aggregator,
    llm,
    LLMTextProcessor(), # optional — converts LLMTextFrame → AggregatedTextFrame
    tts,
    transport.output(),
    assistant_aggregator,
]
worker = PipelineWorker(pipeline, params=PipelineParams(...))
# Access RTVI via worker.rtvi (created internally by WorkerRunner)
@worker.rtvi.event_handler("on_client_ready")
async def on_client_ready(rtvi): ...
```

### 7. Mount order: API routes BEFORE static mount

`app.mount("/client", PipecatPrebuiltUI)` must come AFTER all `/api/offer` routes are registered. Starlette's mount catches all subpaths; if mounted before API routes, `/api/offer` returns 405.

### 8. LiteLLM/Headroom proxy: LLM output silently dropped

When using `OpenAILLMService` pointed at a LiteLLM or Headroom proxy, the proxy transmits response text in `delta.reasoning_content` instead of `delta.content`. Pipecat's `_process_context` only reads `delta.content`, so the LLM generates text (`completion_tokens` counters increment) but no `LLMTextFrame` frames ever reach downstream processors (TTS, BotText, output).

See `voice-ai-pipelines` skill → "LLM" section for the fix (two patches: pydantic model + `_process_context` elif branch).

This affects ALL models through **this specific proxy setup** — DeepSeek, MiniMax, etc. If you can modify the proxy config, merging `reasoning_content` into `content` is cleaner. Otherwise patch pipecat's installed packages.

### 9. `on_client_ready` MUST inject system prompt + queue LLMRunFrame

**Symptom**: The bot stays silent until the user speaks first. Even after the user speaks,
the LLM responds with garbage (off-topic, hallucinates, no system constraints honored).
Server log shows no errors; pipeline appears healthy.

**Root cause**: The `on_client_ready` event handler is registered but does nothing
except log. Without explicitly injecting a system message into the LLM context AND
queuing `LLMRunFrame()`, the LLM runs against an empty context and never opens the
conversation.

**Required pattern** (verified against `pipecat/src/bot_js_client.py:137-145`):

```python
worker, context = build_pipeline(transport=transport, ...)

@worker.rtvi.event_handler("on_client_ready")
async def on_client_ready(rtvi):
    logger.info(f"[{connection.pc_id}] Client ready")
    # 1. Inject system prompt (or developer message) into context
    context.add_message({"role": "system", "content": SYSTEM_PROMPT})
    # 2. Queue LLMRunFrame to trigger LLM generation
    await worker.queue_frames([LLMRunFrame()])
```

**Two-part failure mode if either is missing:**

| Missing | Symptom |
|---|---|
| No `add_message` | LLM responds without persona/constraints — sounds generic or off-character |
| No `queue_frames([LLMRunFrame()])` | Bot silent until user speaks; user speaks and gets unexpected reply |
| Both missing | Silent bot, empty context — looks like the entire voice agent is broken |

**Note on role field**: Use `"system"` (OpenAI-compatible) rather than `"developer"`
(which `bot_js_client.py` uses for GeminiLive). DeepSeek, OpenAI, and LiteLLM proxies
recognize `system`. `developer` is OpenAI's newer role name and may be ignored by
older LiteLLM versions.

**Source**: `bot_js_client.py:137-145` is the canonical reference. Always diff your
`on_client_ready` against this — if yours only logs, copy the pattern.

### 10. DTLS handshake stalls in Cloud browser environments — flush pending messages manually

**Symptom**: Browser connects, ICE negotiates (`iceConnectionState = completed`),
but `connectionState` stays at `"connecting"` forever. After 60 seconds the
connection drops with no error. Server log shows:
```
Client not connected. Queuing app-message.
```
repeatedly for every text message the user sends.

**Root cause** (`pipecat/transports/smallwebrtc/connection.py:656-672`):
```python
def is_connected(self) -> bool:
    if self._last_received_time is None:
        return self._pc.connectionState == "connected"  # ← stuck on "connecting"
    return (time.time() - self._last_received_time) < 3
```

In Cloud browser environments (Browserbase, remote Chromium behind NAT), aiortc's
DTLS handshake completes (ICE `connectionState = "completed"`) but the overall
PeerConnection `connectionState` never reaches `"connected"`. The 3-second ping
window is also unreliable. Text messages via the data channel queue forever.

**Fix — flush the pending queue manually after the transport is created:**

```python
async def _post_transport(transport, worker, connection):
    pending = getattr(connection, "_pending_app_messages", [])
    if pending:
        logger.info(f"[{connection.pc_id}] flushing {len(pending)} queued messages")
        for msg in list(pending):
            await connection._call_event_handler("app-message", msg)
        pending.clear()

run_pipeline = make_run_pipeline(
    on_client_ready=_on_client_ready,
    on_transport_created=_post_transport,
)
```

Wire this through your `make_run_pipeline` factory so every entry point
(PrebuiltUI, vanilla JS, tests) gets the fix.

**When this is safe to do**: after `WorkerRunner.add_workers(worker)` but before
`runner.run()`. The transport exists, no audio has been processed yet, and
`connection._call_event_handler("app-message", msg)` directly invokes the
pipeline-side handler — bypassing `is_connected()` entirely.

## Verification

- `curl http://localhost:8766/` → 307 redirect to /client/
- `curl http://localhost:8766/client/` → 200, HTML with Pipecat UI
- `curl -X POST -d '{}' http://localhost:8766/start` → 200, `{"sessionId": "default", "iceConfig": {...}}`
- Browser: Connect button → status shows Client READY / Agent READY
- Server log after text send should show:
  - `Generating chat from context [{'role': 'user', 'content': '...'}]`
  - `Bot started speaking` (TTS producing audio)
  - PrebuiltUI conversation panel shows both `user` and `assistant` messages

If you see all three of those, the pipeline + RTVI + LLMTextProcessor + observer wiring
is correct. If `Bot started speaking` appears but the conversation panel stays empty,
re-check pitfall #6 (silent RTVI drop).
